# gRPC Remote Control Server for app-qt

**Date:** 2026-05-01  
**Branch:** grpc  
**Status:** Approved

## Goal

Add a gRPC server to `app-qt` that allows remote control of the application via four commands: `Start`, `Stop`, `Pause`, and `Resume`. The server dispatches the appropriate `DecoderControlEvent` into the Qt event loop, mirroring exactly what the GUI does.

---

## Proto Definition

**File:** `src/nfc-app/app-qt/src/main/proto/nfc_control.proto`

```proto
syntax = "proto3";

service RemoteControl {
  rpc Start  (ControlRequest) returns (ControlResponse);
  rpc Stop   (ControlRequest) returns (ControlResponse);
  rpc Pause  (ControlRequest) returns (ControlResponse);
  rpc Resume (ControlRequest) returns (ControlResponse);
}

message ControlRequest {}

message ControlResponse {
  bool   success = 1;
  string message = 2;
  State  state   = 3;
}

enum State {
  UNKNOWN = 0;
  IDLE    = 1;
  RUNNING = 2;
  PAUSED  = 3;
}
```

No `package` declaration — generated classes live in the global namespace (`RemoteControl::Service`).

## New Files

```
src/nfc-app/app-qt/src/main/cpp/rpc/
├── GrpcServer.h      # Owns server lifecycle: start() / stop()
├── GrpcServer.cpp
├── GrpcControl.h     # Implements RemoteControl::Service, dispatches Qt events
└── GrpcControl.cpp
```

### GrpcServer

- Constructor takes port (int) and a pointer to the `QObject` that will receive events (the app's main control object).
- `start()`: builds and launches the gRPC server using `grpc::ServerBuilder`, listening on `0.0.0.0:<port>` with `InsecureServerCredentials`. Registers a `GrpcControl` instance as the service.
- `stop()`: calls `server->Shutdown()` and waits for clean termination.
- Instantiated by `QtApplication` only when `grpc/port > 0` in `QSettings`.

### GrpcControl

- Inherits from `RemoteControl::Service` (generated by protoc).
- Holds a pointer to the target `QObject` to post events to.
- Each RPC implementation:
  1. Creates a `std::shared_ptr<std::promise<DecoderControlEvent::Result>>`.
  2. Constructs a `DecoderControlEvent` with the corresponding command (`Start`, `Stop`, `Pause`, `Resume`) and assigns the promise to its `completion` field.
  3. Posts the event via `QCoreApplication::postEvent()`.
  4. Calls `.get()` on the future with a 5-second timeout. On timeout, returns `ControlResponse{success=false, message="timeout", state=UNKNOWN}` and logs a warning.
  5. Maps the `DecoderControlEvent::Result` to a `ControlResponse` and returns it.

## Changes to Existing Files

### DecoderControlEvent.h

Add a generic completion mechanism usable by any caller (not RPC-specific):

```cpp
struct Result {
    bool    success;
    QString message;
    int     state;   // caller-defined interpretation
};

std::shared_ptr<std::promise<Result>> completion; // nullptr when no one is waiting
```

### QtControl.cpp

In `doStartDecode`, `doStopDecode`, and `doPauseDecode` (and the new `doResumeDecode`): after executing the command, check if `event->completion != nullptr` and resolve it with the real post-execution state.

If `QtControl` already handles `DecoderControlEvent::Resume` (verify during implementation), the existing handler is extended. Otherwise, a new `doResumeDecode` handler is added, dispatching the resume commands to the device task streams.

## Configuration

Read from `QSettings` at startup:

```ini
[grpc]
port=50051
```

In code:
```cpp
QSettings settings;
int port = settings.value("grpc/port", 0).toInt();
if (port > 0)
    grpcServer = new GrpcServer(port, this);
```

If the key is absent or zero, `GrpcServer` is never instantiated and the app behaves identically to today.

## CMake Integration

`src/nfc-app/app-qt/CMakeLists.txt` additions:

1. `find_package(Protobuf CONFIG REQUIRED)`
2. `find_package(gRPC CONFIG REQUIRED)`
3. `add_custom_command` to run `protoc` generating `nfc_control.pb.{cc,h}` and `nfc_control.grpc.pb.{cc,h}` from `nfc_control.proto`.
4. Static library `nfc-control-proto` compiling the generated files, linked against `gRPC::grpc++` and `protobuf::libprotobuf`.
5. `app-qt` target links additionally against `nfc-control-proto`.

The pattern mirrors exactly what `test-rpc/CMakeLists.txt` already does.

## Data Flow

```
gRPC client
  → GrpcControl::Start()             (gRPC thread)
  → creates DecoderControlEvent::Start with completion promise
  → QCoreApplication::postEvent()
  → Qt event loop
  → QtControl::handleEvent()         (Qt main thread)
  → doStartDecode()
  → executes tasks
  → resolves completion promise with Result{success, message, RUNNING}
  → GrpcControl receives future      (gRPC thread)
  → returns ControlResponse to client
```

## Out of Scope

- TLS / authentication (server uses `InsecureServerCredentials`).
- Streaming RPCs.
- Remote configuration of device parameters (separate concern).
- Exposing gRPC to `app-rx` or other applications.
